# Replay-Tests Skill

## Overview

Add additional integration tests from real-life JSON fixtures placed in the crate's `tests/data/replay/` folder, run the test suite, and review or adjust code so that all tests pass. This skill is **optional** and **human-triggered**: run it after you have prepared real-life test data and the migration pipeline has already completed.

**When to use**: After migration, when you have real-world fixture data (e.g. API responses, message payloads) that you want to turn into regression tests and ensure the crate behaves correctly against that data.

## Arguments

```text
$CRATE_NAME   = $ARGUMENTS[0]           # Name of the generated Rust crate (e.g. fleet_api)
$PROJECT_DIR  = $ARGUMENTS[1] OR "."    # Project root (default current directory)
```

Resolve the crate directory using the standard convention `crates/<name>`:

```text
$CRATE_DIR = $PROJECT_DIR/crates/$CRATE_NAME
```

**Examples**:

```prompt
replay-writer fleet_api
replay-writer fleet_api /path/to/project
```

## Prerequisites

- Crate was **generated by the Specify workflow** (e.g. via crate-writer): has `src/`, handlers, and existing `tests/` layout.
- **Real-life test data** has been placed by the user in the crate's replay data folder: `$CRATE_DIR/tests/data/replay/`.
- Fixtures in `tests/data/replay/` follow the [Fixture format](references/fixture-format.md) (TestDef-style or raw payload JSON as used by the crate's MockProvider).
- **Instructions** (optional): An `INSTRUCTIONS.md` file in `tests/data/replay/` providing guidance on sample data loading, MockProvider setup, and assertion patterns. See [Fixture format § Instructions file](references/fixture-format.md).
- **Sample data** (optional): Shared data files in `tests/data/replay/samples/` referenced by fixtures or instructions. These are bulk datasets (not fixtures) that multiple test cases share.

## Crate Layout

The skill assumes the standard layout produced by crate-writer:

```text
$CRATE_DIR/
├── src/
│   ├── lib.rs
│   ├── handler.rs      # (or handlers keyed by domain)
│   └── ...
├── tests/
│   ├── provider.rs     # MockProvider for the crate's provider bounds
│   ├── *.rs            # Integration test modules (e.g. get_vehicles.rs)
│   └── data/
│       └── replay/     # Replay fixtures (JSON) — user places real-life data here
│           ├── INSTRUCTIONS.md  # (optional) special instructions for test generation
│           ├── samples/         # (optional) shared data files referenced by fixtures
│           │   └── *.json
│           └── *.json           # TestDef format or raw payload JSON
├── Cargo.toml
└── ...
```

See [Crate layout](references/crate-layout.md) for details and how fixtures map to handlers and MockProvider.

## Fixture Format and Location

- **Location**: All replay fixtures live under `$CRATE_DIR/tests/data/replay/`.
- **Format**: **TestDef-style JSON** — see [fixture-format.md](references/fixture-format.md) for the expected test fixture format, with some examples in [fixtures.md](examples/fixtures.md)
- **Instructions**: If `$CRATE_DIR/tests/data/replay/INSTRUCTIONS.md` exists, read it **before** generating tests. It provides domain-specific guidance on how to load sample data, construct the MockProvider, and assert on results.
- **Sample data**: Files in `$CRATE_DIR/tests/data/replay/samples/` are shared datasets — not fixtures. They are loaded per the instructions or referenced by fixtures via `@samples/` paths (see [fixture-format.md § Setup block](references/fixture-format.md)).

When adding new tests from fixtures, follow the crate's existing test style (same MockProvider, same `include_bytes!` or file-based loading, same assertion patterns).

## Execution Steps

1. **Inspect crate, instructions, and fixtures**
   - Read `$CRATE_DIR/src/` to identify handlers, input/output types, and provider bounds.
   - Read `$CRATE_DIR/tests/` to see existing test modules and how they use `tests/data/replay/`.
   - If `$CRATE_DIR/tests/data/replay/INSTRUCTIONS.md` exists, read it first. Use its guidance to determine how sample data is loaded, how the MockProvider should be constructed, and any special assertion logic.
   - List `$CRATE_DIR/tests/data/replay/samples/` for shared data files. These are **not** fixtures — do not generate tests from them directly. They are referenced by fixtures or loaded per the instructions.
   - List `$CRATE_DIR/tests/data/replay/*.json` and classify each as TestDef-style or raw payload.

2. **Add or extend tests**
   - For each fixture in `tests/data/replay/` that is not yet covered by a test (or that represents a new scenario), add or extend integration tests that:
     - Load the fixture from `tests/data/replay/` (e.g. `include_bytes!("data/replay/<file>.json")` or open file).
     - If a fixture contains a `setup` block, use it to configure the MockProvider before invoking the handler. The `setup` block may reference `@samples/` files for bulk data, specify cache seeding, or override config values (see [fixture-format.md § Setup block](references/fixture-format.md)).
     - If no `setup` block is present, fall back to instructions in `INSTRUCTIONS.md`, then to the crate's existing test patterns.
     - Use the crate's existing `provider.rs` MockProvider and request/response types. Some tests might require a custom MockProvider.
     - Invoke the appropriate handler and assert on response or side effects.
   - Preserve existing test style and naming. Do not remove or weaken existing tests.

3. **Run tests**
   - Run `cargo test` in `$CRATE_DIR`.
   - If any test fails, proceed to step 4.

4. **Review and fix so tests pass**
   - Analyze failures (wrong assertions, missing MockProvider setup, type mismatches, fixture shape vs handler expectations).
   - Propose and apply minimal code or test changes so that:
     - All tests pass.
     - Production code remains compliant with guardrails (no `unwrap()`/`expect()` in production, no forbidden crates/APIs).
   - Re-run `cargo test` until all tests pass.

5. **Summarize**
   - Report: tests added or updated, failures fixed, and final `cargo test` result.

## Authority and Conventions

- **This SKILL.md** — generation and review rules.
- **Existing crate code and tests** — handler signatures, MockProvider usage, and test patterns.
- **references/** — [fixture-format.md](references/fixture-format.md), [crate-layout.md](references/crate-layout.md).
- **examples/** - [handler.md](examples/handler.md), [fixtures.md](examples/fixtures.md), [tests.md](examples/tests.md)
- **Omnia** — handler pattern, provider traits, error handling (see `.cursor/rules/project.mdc` and `.cursor/rules/omnia.mdc`).

Do not change handler semantics or add production code beyond what is strictly required to make tests pass. Prefer fixing test setup or assertions over changing library code unless the failure indicates a real bug.

## Verification

Before completing:

- [ ] All fixtures under `tests/data/replay/` are either used by existing tests or have new tests added.
- [ ] `cargo test` in `$CRATE_DIR` passes.
- [ ] `cargo clippy -- -D warnings` passes (recommended).
- [ ] No new `unwrap()` or `expect()` in production code; test code may use them where appropriate.
